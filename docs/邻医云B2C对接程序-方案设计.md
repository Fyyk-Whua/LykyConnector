## 一、接口文档解析

### 1.1 平台背景

**邻医云 B2C 开放平台**由邻医快药拥有并独立经营，是基于各大三方平台（美团、淘宝闪购、京东、拼多多、抖音、百度健康、京东药急送等）和各品牌 ERP 系统对接业务流程，为**药店**以及服务于此类商家的**第三方软件服务商（ISV）**提供系统对接服务的平台。

核心目标：通过接口对接，让药店 ERP 自主实现订单同步、库存同步、价格同步、消息推送等功能，去除人工操作成本和出错率，实现高效运营。

### 1.2 接入协议

| 项目 | 说明 |
|------|------|
| 接口地址 | `https://store-api.lyky.cn/` |
| 编码格式 | UTF-8 |
| 调用方式 | POST |
| 数据格式 | `application/x-www-form-urlencoded`（Header 强制校验） |
| 开发者中心 | https://b2c-open.lyky.cn |
| 沙箱环境 | 支持（可模拟消息推送到开发者配置的推送地址） |

**系统级参数**（所有接口必传）：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `app_id` | string | 创建应用时生成的应用 ID |
| `timestamp` | long | 10 位秒级时间戳，作为时间戳校验依据 |
| `signature` | string | 按签名规则计算的签名结果 |
| `version` | string | 接口版本，默认 `v1` |

**签名规则**：
1. 将所有系统级参数 + `app_secret`（signature 除外）按参数名字母顺序排序
2. 将排序后的参数**值**顺序拼接成字符串
3. 对拼接字符串先 **MD5** 加密，再 **Base64** 编码，得到 signature

```
示例：app_id + app_secret + timestamp + version → MD5 → Base64
```

## 二、需求分析

### 2.1 业务背景与问题

药店在美团、京东、淘宝闪购、拼多多、抖音等多个 O2O/电商平台开店售药。这些平台的订单通过**邻医云开放平台**统一汇聚后，需要实时推送到药店的 **ERP/收银系统**进行出库、销账；同时药店 ERP 中的库存、价格、货位变化也需要实时同步回各平台。

当前痛点：
- **人工搬单**：店员手动在各平台后台抄录订单，易漏单、错单，效率低
- **库存不同步**：ERP 卖出后未及时同步平台，导致超卖退款、客诉
- **医药合规风险**：处方单、追溯码若未及时处理，违反 GSP 监管要求
- **多平台分散**：9 个平台分散管理，无法统一监控

### 2.2 系统定位

本程序是一个**本地化运行的对接中间件（网关）**，部署在药店 ERP 电脑或门店服务器上，承担"邻医云开放平台 ⇄ 药店 ERP"之间的双向数据桥梁：

```
┌─────────────┐    推送订单/售后/发货/退货/追溯码    ┌──────────────────┐    落库/出库    ┌──────────┐
│  邻医云开放   │ ──────────────────────────────────▶│  本对接程序(网关)  │ ─────────────▶│ 药店ERP  │
│  平台(9平台)  │ ◀──────────────────────────────────│  (本地常驻运行)    │ ◀─────────────│ 收银系统  │
└─────────────┘    同步库存/价格/货位/发货/面单      └──────────────────┘   查询/回传     └──────────┘
```

- **作为服务端**：内置 HTTP 服务，暴露回调地址，接收平台 7 类推送消息 → 验签 → 转发 ERP
- **作为客户端**：调用平台 7 个主动接口，把 ERP 的库存/价格/货位/发货/面单同步上去
- **作为桌面程序**：提供配置、监控、日志、告警的可视化管理界面

### 2.3 用户角色

| 角色 | 说明 | 核心诉求 |
|------|------|----------|
| 药店店长/管理员 | 配置与监控程序运行 | 配置简单、运行稳定、一目了然 |
| 药店店员/收银员 | 日常使用（被动） | 程序不可被误关、订单自动到位 |
| ISV 实施人员 | 部署与对接调试 | 部署便捷、日志清晰、沙箱可测 |
| 运维/IT | 维护与排障 | 远程诊断、自动恢复、告警通知 |

### 2.4 功能性需求

#### FR-1 订单消息接收与转发（核心）
- 接收邻医云推送的 7 类消息，验签后落库并转发药店 ERP
- 支持原始订单、售后单、发货通知、退货通知、出库、追溯码更新
- 处方单特殊标记与提醒
- 推送失败自动重试 + 告警

#### FR-2 主动数据同步
- 库存同步：ERP 库存变化时调用 `update-stock-one` 同步平台
- 价格同步：成本价变化时调用 `update-cost-price-one`
- 货位同步：调用 `update-cargo-one`
- 批号同步：订单发货前调用 `order-sync/batch` 同步批号
- 电子面单：调用 `get-way-bill` 获取（≤50 单批量）、`cancel-way-bill` 取消
- 发货同步：调用 `delivery` 回传发货信息（含快递公司枚举映射）

#### FR-3 配置管理
- 应用凭证配置（app_id / app_secret，加密存储）
- 回调地址配置（本机回调端口 / 公网映射地址）
- ERP 对接配置（ERP 接口地址、鉴权、字段映射）
- 平台店铺绑定（store_id / merchant_id）

#### FR-4 监控与告警
- 实时运行状态（连接、心跳、服务在线）
- 订单接收/同步实时流水
- 异常告警（推送失败、签名错误、ERP 不通、库存同步失败店铺列表）
- 告警方式：界面弹窗 + 声音 + 企业微信/钉钉机器人

#### FR-5 日志与审计
- 全量接口调用日志（请求/响应/耗时）
- 推送消息留存（可重放）
- 操作审计日志
- 日志按天滚动，可导出

#### FR-6 数据看板
- 今日订单量、各平台占比
- 同步成功率、失败数
- 库存同步失败店铺明细
- 待处理处方单数

### 2.5 非功能性需求（重点）

#### NFR-1 进程保护（用户重点要求）
- 主程序异常崩溃后**自动重启**
- 防止被任务管理器结束（看门狗守护 + 双进程互守）
- 防止系统资源不足被 OOM 杀死（内存监控 + 自动回收）

#### NFR-2 防止用户随意退出（用户重点要求）
- 关闭按钮需**管理员密码**确认（普通店员无法退出）
- 系统托盘常驻，最小化到托盘而非退出
- 屏蔽 Alt+F4、任务栏右键关闭
- 退出后看门狗自动拉起

#### NFR-3 开机自动运行（用户重点要求）
- 系统开机/用户登录后自动启动
- 启动后自动恢复上次运行状态
- 支持延迟启动（等待网络就绪）

#### NFR-4 高可用与可靠性
- 7×24 稳定运行，医药订单不丢失
- 本地消息队列持久化，断网/ERP 不通时消息不丢
- 断网自动重连，积压消息补推
- 推送幂等，防止重复处理

#### NFR-5 性能
- 推送消息验签 + 转发 < 200ms
- 库存同步并发处理，单店 < 1s
- 界面操作响应 < 300ms

#### NFR-6 安全
- app_secret 加密存储（DPAPI/AES）
- 回调地址验签防伪造
- 通信全程 HTTPS
- 敏感信息（收货人隐私）脱敏显示
- 操作权限分级

#### NFR-7 可维护
- 一键安装/卸载
- 配置导入导出
- 远程日志查看
- 自动更新（可选）

### 2.6 约束与假设
- 部署环境：Windows 7/10/11 药店收银电脑或门店服务器
- 网络环境：需稳定互联网（对接开放平台）+ 局域网（对接 ERP）
- 药店 ERP 需提供接收订单的本地接口（HTTP/数据库两种模式可选）
- 一个程序可对接多个店铺（多 store_id）

## 三、开发方案

### 3.1 技术选型

**推荐方案：C# .NET 8 + WPF + Kestrel**

| 层 | 技术 | 选型理由 |
|----|------|----------|
| 桌面 UI | **WPF（MVVM）** | Windows 原生桌面，界面精美，支持数据绑定/动画；成熟稳定 |
| 内置 HTTP 服务 | **Kestrel（ASP.NET Core）** | .NET 内置高性能 Web 服务器，可宿主于桌面进程接收推送 |
| HTTP 客户端 | **HttpClient + Polly** | 调用开放平台接口，Polly 提供重试/熔断/超时 |
| 消息队列 | **SQLite + 本地队列** | 轻量持久化，断电不丢消息，无需额外服务 |
| 日志 | **Serilog**（文件 + 滚动） | 结构化日志，按天滚动 |
| 进程保护 | **Windows Service + 看门狗进程** | 服务级守护，开机自启，崩溃自拉起 |
| 开机自启 | **注册表 Run + 任务计划程序** | 双保险自启 |
| 加密 | **DPAPI**（密钥）+ AES | Windows 原生密钥保护 |
| 告警通知 | **企业微信/钉钉 Webhook** | 异常即时推送 |
| 打包 | **Inno Setup / MSIX** | 一键安装 |

> **为什么选 C# .NET 而非 Electron/Python？**
> - 进程保护、开机自启、Windows 服务集成是 .NET 的强项，原生 API 调用成熟
> - WPF 桌面体验与性能优于 Electron（医药门店老电脑较多，Electron 内存占用高）
> - Kestrel 内嵌 HTTP 服务接收推送，单进程即可完成双向网关
> - 7×24 稳定运行，GC 与内存管理可靠

### 3.3 模块设计

#### 模块 1：推送接收服务（PushReceiver）
- Kestrel 监听本地端口（默认 8686），暴露 7 个推送回调路由
- 收到推送 → 验签（MD5+Base64）→ 写入本地队列 → 立即返回 `{"data":"ok"}`
- 异步消费队列 → 转发 ERP（HTTP 模式调用 ERP 接口 / DB 模式写库）
- ERP 不通时消息持久保留，定时重试

#### 模块 2：同步调度引擎（SyncEngine）
- 监听 ERP 库存/价格变化（ERP 主动推送 / 定时轮询 / 数据库触发）
- 调用开放平台 7 个主动接口
- 快递公司枚举映射表
- 电子面单批量获取（≤50）与打印数据解析
- Polly 重试策略（指数退避，最多 3 次）

#### 模块 3：签名服务（SignService）
- 统一封装签名计算与验签
- 主动调用：参数排序 → 值拼接 → MD5 → Base64
- 推送验签：用同样算法校验 signature 是否一致

#### 模块 4：本地持久化队列（MessageQueue）
- 基于 SQLite 的持久化消息队列
- 消息状态：待处理 / 处理中 / 成功 / 失败 / 死信
- 断电重启后自动恢复未处理消息
- 死信队列人工介入

#### 模块 5：告警通知（AlertService）
- 触发条件：推送验签失败、ERP 连续不通、同步失败超阈值、程序异常
- 通知渠道：UI 弹窗 + 系统声音 + 企业微信/钉钉 Webhook
- 告警去重与分级（警告/严重）

#### 模块 6：桌面 UI（WPF）
- 详见第四节《界面方案》

### 3.4 进程保护方案（重点）

采用**"主程序 + 看门狗"双进程互守**架构：

```
┌─────────────────────────────────────────┐
│           Windows 任务计划/注册表          │  ← 开机自启入口
│   启动 Watchdog.exe（最高优先级）          │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│         Watchdog.exe（守护进程）          │
│  • 每 3 秒检测主程序进程是否存活           │
│  • 主程序退出/崩溃 → 立即重新拉起          │
│  • 自身异常时由主程序反向拉起（互守）       │
│  • 以 Windows Service 方式注册，防被杀     │
└──────────────────┬──────────────────────┘
                   ▼ 拉起
┌─────────────────────────────────────────┐
│      MainApp.exe（主程序 + UI）           │
│  • 反向监控 Watchdog，挂了则拉起           │
│  • 全局异常捕获 → 自我重启                │
│  • 内存超阈值自动 GC 回收                  │
└─────────────────────────────────────────┘
```

**具体措施**：
1. **双进程互守**：主程序与看门狗互相监控，任一挂掉对方立即拉起
2. **Windows 服务化**：看门狗注册为系统服务，权限独立，普通用户无法从任务管理器结束
3. **全局异常捕获**：`App.DispatcherUnhandledException` + `AppDomain.UnhandledException` + `TaskScheduler.UnobservedTaskException` 三重捕获，崩溃前通知看门狗
4. **心跳机制**：主程序每 5 秒写心跳文件，看门狗检测心跳超时则判定假死并重启
5. **防 OOM**：监控内存占用，超 800MB 主动回收或优雅重启

### 3.5 防止用户随意退出方案（重点）

| 防护点 | 措施 |
|--------|------|
| 关闭按钮 | 标题栏关闭按钮点击 → 弹出"退出需管理员密码"对话框，验证通过才退出 |
| Alt+F4 | 拦截系统按键，需密码确认 |
| 任务栏右键关闭 | 隐藏任务栏图标，仅托盘常驻；右键菜单无"关闭"项 |
| 任务管理器结束 | 看门狗为系统服务，普通用户无权限结束；主程序被结束后看门狗自动拉起 |
| 最小化 | 最小化到系统托盘，程序继续运行 |
| 退出后 | 即便密码退出，看门狗仍会 3 秒内重新拉起（除非通过"维护模式"正式停止） |
| 正式停止 | 需"管理员密码 + 维护模式开关"双重确认，仅 IT 人员操作 |

**正式退出流程**（仅 IT 维护用）：
1. 托盘菜单 → "维护模式"（需密码）
2. 维护模式下点击"停止服务" → 看门狗停止守护
3. 主程序正常退出

### 3.6 开机自动运行方案（重点）

**双保险自启**：

1. **注册表自启**（用户级）
   - `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 写入 Watchdog.exe 路径
   - 用户登录即启动

2. **任务计划程序**（系统级，更可靠）
   - 创建任务"LykyConnector"，触发器"登录时"
   - 以最高权限运行，勾选"即使电池供电也运行"
   - 可配置延迟 30 秒（等待网络就绪）

3. **网络就绪检测**
   - 启动后检测网络连通性，未就绪则每 10 秒重试
   - 网络就绪后自动连接开放平台

### 3.7 安全方案

- **密钥保护**：app_secret 使用 Windows DPAPI 加密存储，绑定本机
- **通信加密**：与开放平台全程 HTTPS；与 ERP 局域网可配置 HTTPS
- **验签防伪**：所有推送消息强制验签，签名不符拒绝并告警
- **隐私脱敏**：收货人姓名/手机/地址界面默认脱敏，需权限查看明文
- **权限分级**：管理员（配置/退出）、操作员（仅查看）、维护（IT 模式）
- **审计日志**：所有配置变更、退出操作、敏感数据查看均记录审计

### 3.8 部署方案

- **安装包**：Inno Setup 制作单文件安装包（约 60MB，含 .NET 运行时）
- **安装流程**：一键安装 → 自动注册服务/任务计划 → 填写 app_id/app_secret → 测试连通 → 完成
- **首次配置向导**：引导配置凭证、回调地址、ERP 对接、店铺绑定
- **沙箱测试**：内置沙箱模式开关，可模拟推送测试

### 3.9 开发阶段划分

| 阶段 | 内容 | 产出 |
|------|------|------|
| P1 基础框架 | 项目骨架、签名服务、HTTP 客户端、Kestrel 推收 | 可收发消息的最小网关 |
| P2 持久化与队列 | SQLite 队列、重试、死信、断网恢复 | 可靠消息通道 |
| P3 主动同步 | 7 个主动接口封装、ERP 适配器、调度引擎 | 完整双向对接 |
| P4 桌面 UI | 监控/配置/日志/看板界面 | 可用桌面程序 |
| P5 进程保护 | 看门狗、防退出、开机自启 | 生产级稳定性 |
| P6 测试与打包 | 沙箱联调、安装包、部署文档 | 可交付产品 |

## 四、界面方案

### 4.1 设计原则

- **信息密度合理**：监控数据一目了然，关键异常醒目
- **医药行业气质**：专业、稳重，主色采用医药蓝/青绿，状态色清晰
- **托盘优先**：主窗口可关闭，但程序常驻托盘
- **响应迅速**：实时数据流采用虚拟化列表，避免卡顿
- **权限可见**：不同角色看到不同操作项

### 4.2 整体布局

采用**左侧导航 + 顶部状态栏 + 主内容区**的经典桌面布局：

```
┌──────────────────────────────────────────────────────────────┐
│  顶栏：[● 运行中]  邻医云对接服务   |  店铺:3  今日订单:128  ⚙ 管理  ▾  │
├────────┬─────────────────────────────────────────────────────┤
│        │                                                     │
│  导航   │                  主内容区                            │
│        │                                                     │
│ 📊看板  │                                                     │
│ 📦订单流 │                                                     │
│ 🔄同步  │                                                     │
│ 📋日志  │                                                     │
│ ⚠告警  │                                                     │
│ ⚙配置  │                                                     │
│        │                                                     │
├────────┴─────────────────────────────────────────────────────┤
│  状态栏：网络:正常 | ERP:已连接 | 队列:0 | CPU:3% | 内存:210MB  v1.0 │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 各页面设计

#### 页面 1：运行看板（首页）
- **顶部 KPI 卡片**：今日订单数、同步成功率、待处理处方单、告警数
- **平台订单占比**：环形图（京东/美团/天猫/拼多多/抖音...）
- **同步趋势**：近 7 天订单量折线图
- **实时动态**：最近 20 条订单/同步事件滚动流
- **失败店铺榜**：库存同步失败 Top 店铺列表

#### 页面 2：订单流水
- **列表区**：时间、平台、订单号、收货人(脱敏)、金额、状态、处方标记
- **筛选**：按平台、状态、日期、处方单筛选
- **详情抽屉**：点击查看完整订单（商品明细、地址、批号、追溯码）
- **操作**：手动重推 ERP、标记已处理
- **处方单高亮**：处方单行用特殊色标记

#### 页面 3：同步管理
- **库存同步**：SKU 列表、当前库存、同步状态、手动同步按钮
- **价格同步**：成本价变更记录
- **电子面单**：批量获取（≤50）、面单预览、取消面单
- **发货同步**：待发货订单、快递公司选择、回传发货
- **批号同步**：订单批号绑定
- 每项支持手动触发 + 自动模式开关

#### 页面 4：日志中心
- **接口日志**：请求/响应/耗时/状态，可按接口类型筛选
- **推送日志**：7 类推送消息原文留存，可重放
- **系统日志**：程序运行、异常、重启记录
- **审计日志**：配置变更、退出操作、敏感查看
- 支持导出、按天查看、关键字搜索

#### 页面 5：告警中心
- **实时告警列表**：级别、时间、内容、状态（未处理/已处理）
- **告警规则配置**：阈值设置（连续失败次数等）
- **通知渠道配置**：企业微信/钉钉 Webhook
- 声音开关、免打扰时段

#### 页面 6：系统配置
- **应用凭证**：app_id / app_secret（加密显示）
- **回调设置**：本机端口、公网映射地址（展示给开放后台填写）
- **ERP 对接**：模式(HTTP/DB)、地址、鉴权、字段映射
- **店铺绑定**：store_id / merchant_id 列表管理
- **运行策略**：开机自启、进程保护、自动重连间隔
- **沙箱模式**：切换正式/沙箱环境
- **维护模式**：IT 专用（需密码）

#### 页面 7：系统托盘
- 托盘图标显示运行状态（绿=正常/黄=告警/红=异常）
- 右键菜单：显示主界面、暂停同步（需密码）、维护模式、关于
- 气泡通知：新订单、异常告警

### 4.4 交互细节

- **退出保护**：点关闭按钮 → 弹窗"退出服务将停止订单同步，确认退出？请输入管理员密码" → 验证通过才退出
- **异常醒目**：告警用红/橙色，顶部状态栏实时反映
- **处方单提醒**：收到处方单时托盘气泡 + 轻提示音
- **数据刷新**：列表自动刷新（可配置间隔），手动刷新按钮
- **暗色模式**：支持浅色/深色/跟随系统（夜间值守友好）

## 附录：技术风险与对策

| 风险 | 对策 |
|------|------|
| 开放平台接口变更 | 版本化封装，接口适配层隔离变更 |
| 药店 ERP 接口不标准 | 提供 HTTP/DB 双模式适配器 + 字段映射配置 |

| 网络不稳定 | 持久化队列 + 断网重连 + 积压补推 |
| 处方药合规 | 处方单标记提醒 + 追溯码同步 + 操作审计 |
